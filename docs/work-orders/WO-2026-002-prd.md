# AI Work Order

**ID:** WO-2026-002
**Sprint:** Phase 1 / Week 1 — PRD Planning
**Primary Agent:** John / `bmad-agent-pm`
**Supporting Agents:** Mary / `bmad-agent-analyst` for handoff context; advisory `skeptic` and `product-manager` for guardrail review if PRD scope drifts
**Status:** Ready

## Goal

Create the BMAD Product Requirements Document for SpeakFit English from the approved Product Brief, converting the Discovery boundary into precise MVP requirements that John can hand off to UX, Architecture, Epics/Stories, and Sprint Planning.

## Input Artifacts

- `_bmad-output/planning-artifacts/product-brief.md` — approved Discovery source of truth for PRD creation.
- `.cursor/rules/project-context.md` — goal hierarchy, working mode, desktop-first MVP focus, and BMAD boundaries.
- `docs/decisions.md` — committed product/process/stack decisions and approval trail.
- `BMAD_SPEAKFIT_WORKFLOW.md` — Solo PM + AI agent operating playbook and phase boundaries.
- `docs/INDEX.md` — context loading and documentation maintenance policy.
- `PLANNING_QUESTIONS.md` — archived planning canvas; use only as fallback detail when the Product Brief is insufficient.
- `IDEA_BRIEF.md` — vision/persona color only; do not treat as source of truth.

## Expected Output

- `_bmad-output/planning-artifacts/prd.md`
- PRD handoff summary in chat for Solo PM review.
- Explicit list of PRD assumptions, open questions, MVP cuts, and risks to add or update in `docs/raid.md`.
- Clear downstream handoff notes for UX Spec, Architecture, Epics/Stories, and Sprint Planning.

## Acceptance Criteria

- [ ] PRD uses the approved Product Brief as the primary source of truth and does not reopen already approved Discovery positioning unless a contradiction blocks requirements clarity.
- [ ] PRD preserves the MVP center: `Mission -> Warm-up -> Speak/Shadow -> Feedback -> Retry -> Save -> Progress`.
- [ ] PRD defines MVP requirements precisely enough for UX and Architecture to proceed without inventing product scope.
- [ ] PRD keeps the first learning experience optimized for desktop/laptop focused study sessions; mobile responsive support is secondary and must not drive MVP UX complexity.
- [ ] PRD includes clear functional requirements for onboarding, daily mission flow, listening/shadowing, recording/replay, feedback, retry, saved phrases, lightweight progress, and AI/TTS/STT usage controls.
- [ ] PRD distinguishes must-have MVP requirements from post-MVP backlog ideas, especially around mobile polish, broad course library, real-time simulator, deep pronunciation scoring, payments, social/community, and heavy gamification.
- [ ] PRD defines non-functional requirements for privacy-sensitive speaking data, cost/quota control, reliability of the practice loop, and portfolio/interview-defendable engineering evidence.
- [ ] PRD includes measurable success signals for 7-day validation: mission completion, speaking attempts, retry behavior, saved phrase usage, confidence before/after, and feedback usefulness.
- [ ] PRD captures risky assumptions and open questions explicitly enough to feed `docs/raid.md`.
- [ ] PRD does not specify implementation-level architecture, database schema, API contracts, UI component design, or sprint stories; those belong to later BMAD phases.

## Constraints

- Scope: PRD creation only; no UX spec, architecture design, epics/stories, sprint plan, or implementation.
- Format: Markdown BMAD planning artifact suitable for Solo PM approval.
- Length: Complete but reviewable; prefer structured requirement groups and clear priority labels over exhaustive narrative.
- Language: English for the output artifact unless the Solo PM explicitly asks otherwise.
- Source discipline: Start from the Product Brief and decision log. Do not pull older brainstorm content into PRD unless it directly supports an approved Product Brief requirement.
- MVP boundary: Validate the daily voice coaching loop, not the full 60-day product vision.
- Desktop-first note: Optimize focused learning on laptop/desktop. Mobile should remain usable/responsive where reasonable, but must not introduce phone-first interaction patterns, extra navigation complexity, or MVP feature expansion.
- Non-goals: No code, no data model, no API design, no UI mockups, no architecture locking, no payments, no social features, no generic English curriculum expansion.

## Guardrails from Skeptic Review

- The biggest failure mode is not technical feasibility; it is daily behavior change. PRD must make the daily loop small, obvious, and repeatable rather than feature-rich.
- Avoid pretending AI feedback automatically creates learning. Require feedback to be concise, actionable, and tied to a retry.
- Do not let “personalized coach” become a vague promise. Personalization in MVP should be limited to goal, level, professional track, saved phrases, repeated mistakes if available, and current mission context.
- Content quality is likely a bigger bottleneck than UI. PRD should require enough curated mission structure and sample answers for the first test segment instead of relying on open-ended AI generation alone.
- Desktop-first is a deliberate focus choice, not an excuse to ignore basic responsive behavior. But mobile polish must not become a hidden MVP project.
- Protect against portfolio over-engineering: PRD may require evidence and auditability, but must not load Phase 4 backend showcase features into MVP.

## Guardrails from Product Manager Review

- Keep the primary persona specific: Vietnamese A2-high/B1/B1+ professionals preparing for interview or workplace speaking, not “all English learners.”
- Requirements should map to the user job: “I know what to practice today, I speak, I get useful feedback, I retry, and I can see progress.”
- MVP should test retention and speaking behavior before testing monetization, community, content scale, or advanced simulation.
- Define success metrics around behavior and learning value, not vanity usage: completed missions, recordings/submissions, retries after feedback, saved/reused phrases, confidence delta, and return behavior.
- Treat the first 7-14 day validation loop as a product requirement input: PRD should make it easy to dogfood and inspect whether the loop works.
- Every requirement should pass this cut test: “If this is removed, can we still validate the daily voice coaching loop?” If yes, move it to backlog unless the Solo PM explicitly promotes it.

## Deadline

Phase 1 / Day 2.

## Risk Hints

- John may over-expand PRD into a full English learning platform because the broader 60-day vision is compelling.
- John may under-spec validation and feedback quality, leaving UX/Architecture to invent critical behavior-loop details.
- John may treat mobile as equal priority and accidentally drive phone-first UX complexity against the current MVP direction.
- John may over-lock architecture because the stack direction is already reconfirmed; PRD should describe needs and constraints, not design the backend.
- John may omit portfolio/orchestration success signals even though they are first-class project goals.

## Success Signal

The Solo PM can approve `_bmad-output/planning-artifacts/prd.md` as the requirements source for UX, Architecture, Epics/Stories, and Sprint Planning, with a desktop-first MVP that clearly validates the daily voice coaching loop without drifting into a generic English app or portfolio-driven overbuild.
